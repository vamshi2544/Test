What it does

Receives the high-level goal, breaks it into subtasks, decides which agent handles what, and stitches the final output together.

Responsibilities

  • Task decomposition into steps
  • Agent selection and routing
  • Result aggregation
  • Error handling and retries

Frameworks

LangGraph (complex conditional flows), CrewAI (role-based), AutoGen (conversational), n8n (visual low-code).

LLM — The Brain
Layer: Intelligence · Analogy: The Java runtime — everything runs through it

What it is

The large language model is the reasoning, planning, decision-making, and generation engine. Not a tool — it IS the agent intelligence.

What it decides

  • Which tool to call next
  • Whether task is complete
  • How to format output
  • When to ask for clarification
  • How to recover from errors

Key insight

Swap the LLM like a database driver. The agent framework is the app layer; the LLM is the engine. Same code, different model = different behavior.

Prompt / Context Manager
Layer: Orchestration · The most underrated component in any agent system

What it does

Assembles the final prompt that gets sent to the LLM. Pulls from memory, retrieved RAG chunks, system instructions, conversation history, and the current user input — in the right order.

Why it matters

  • Context window is finite — you can't include everything
  • Order and formatting of context affects output quality
  • Lost-in-the-middle problem — LLMs miss info in the middle
  • Prompt compression and summarization needed at scale

Strategies

  • Sliding window — drop oldest messages
  • Summarize — compress old turns into summary
  • Selective retrieval — only inject relevant memory chunks
  • Token budgeting — allocate tokens per section explicitly

ReAct Agent
Pattern: Reason + Act · Build this before any multi-agent work

The loop

Think → Act (call tool) → Observe (read result) → Think again → repeat until goal is reached.

Why it matters

Every complex multi-agent system is built from ReAct agents. You cannot understand multi-agent without mastering this first.

Where to build

  • LangChain AgentExecutor — fastest start
  • LangGraph — more control, production-grade
  • Raw function calling via Claude or OpenAI API
Specialized Sub-Agents
Pattern: Role-based decomposition · Analogy: Microservices

The idea

Give each agent a specific role, persona, and toolset. Like microservices — each does one thing well with a clear responsibility boundary.

Common roles

  • Researcher — web search, retrieval
  • Coder — writes and executes code
  • Critic / QA — validates outputs
  • Planner — decomposes tasks
  • Executor — real-world actions

Frameworks

CrewAI has the best DX for role-based agents. AutoGen works well for conversational patterns between agents.

Router Agent
Pattern: Intent classification + routing · Analogy: Apigee routing rules powered by LLM

What it does

Classifies the incoming task and routes it to the correct specialist agent or workflow branch.

Example

Message arrives → Router: code task? → CodeAgent. Data question? → AnalystAgent. General? → ReActAgent.

Implementation

LangGraph conditional edges for routing between nodes, or a lightweight classifier LLM call before the main agent runs.

Tool Layer
Layer: Actions · Analogy: External service integrations in Spring Boot

What tools are

Functions the LLM can invoke. You define the function and its description. The LLM decides when and which one to call.

Tool types

  • Web search (Tavily, Bing API)
  • Code execution (Python REPL)
  • Database queries (SQL tools)
  • File read / write operations
  • REST API calls to external services
  • Browser automation (Playwright)

Tool design best practices

  • Write clear, specific descriptions — LLM reads these to decide when to call
  • Handle errors gracefully — return structured error messages, not exceptions
  • Keep tools atomic and composable
  • MCP (Model Context Protocol) — emerging standard for tool interfaces
Memory System
Layer: Context · Analogy: Session state + persistent database combined

Types

  • In-context / short-term — conversation window
  • Semantic / long-term — embedded in vector DB
  • Episodic — logs of past agent run history
  • Procedural — how to do things (system prompts)
  • Entity memory — facts about specific people/things

Architectures

  • Mem0 — managed memory API layer
  • MemGPT — OS-inspired memory paging
  • LangGraph checkpointing — built-in state persistence

Why it matters

Without memory every run starts from zero. Memory is what makes agents useful across sessions and what separates prototypes from real systems.

RAG Pipeline
Layer: Knowledge · Retrieval-Augmented Generation

Pipeline steps

  • 1. Ingest — load docs, split into chunks
  • 2. Embed — convert to vector representations
  • 3. Store — save in vector DB with metadata
  • 4. Query — embed question, find nearest chunks
  • 5. Augment — inject into prompt
  • 6. Generate — grounded response

Vector DB internals

  • HNSW — approximate nearest neighbor indexing
  • ANN — approximate nearest neighbor search
  • Metadata filtering — filter before or after search
  • Hybrid search — keyword + semantic combined

How agents use RAG

Agents call RAG as a tool: search_knowledge_base(query). The agent decides when to retrieve vs when to use existing context.

Output Parser / Structured Output
Layer: Output · Sits between every LLM response and the rest of the system

What it does

Takes the raw LLM text response and validates, parses, and enforces a schema before it's used by the next agent, tool, or returned to the user.

Patterns

  • Pydantic models — define expected output shape
  • JSON mode — force LLM to respond in JSON
  • Instructor library — structured outputs via function calling
  • Retry on malformed output — re-prompt with error feedback
  • Output guardrails — schema validation post-generation

Why critical

Multi-agent systems pass outputs between agents. If Agent A returns malformed output, Agent B fails. Structured outputs are the contract between agents — like typed interfaces in Java.

Human-in-the-Loop (HITL)
Layer: Control · Critical for any high-stakes or irreversible action

What it is

Checkpoints in the agent workflow where a human must approve, review, or provide input before the agent proceeds — especially before irreversible actions.

Patterns

  • Interrupt / resume — pause graph, wait for approval
  • Approval gate — agent proposes, human approves action
  • Feedback loop — human corrects agent output mid-run
  • Confidence threshold — auto-proceed above threshold, pause below

Implementation

LangGraph has built-in interrupt/resume support. For simpler flows, a webhook callback or UI approval step before the agent continues works well.

State Management
Layer: Infra · Analogy: Distributed session state + task graph

What state tracks

  • What has been done / in progress
  • Intermediate results per agent
  • Conversation and run history
  • Current plan and next steps
  • Error state and retry count

LangGraph approach

State is a typed dict passed through graph nodes. Each node reads and writes state. The graph controls flow. Checkpointing enables pause/resume.

Persistence

Serialize state to Redis or Postgres between steps so long-running agents can survive restarts and resume from exactly where they stopped.

Guardrails + AI Security
Layer: Infra · Non-negotiable for any production system

Guardrails coverage

  • Input validation — schema enforcement
  • Output validation — hallucination checks
  • PII detection and masking
  • Policy and compliance enforcement
  • LLM-as-judge pattern for output quality

AI Security threats

  • Prompt injection — malicious input hijacks agent
  • Jailbreaking — bypassing model safety guardrails
  • Data poisoning — corrupting training or RAG data
  • Adversarial inputs — crafted inputs causing wrong outputs
  • Indirect injection via tool results

Architecture position

Wraps every LLM call as middleware — filter on both the request in and the response out. Treat it like a security layer, not an afterthought.

Observability + Cost Management
Layer: Infra · Analogy: Splunk + New Relic + billing dashboard for agent runs

What to observe

  • Every LLM call — prompt, response, tokens, latency
  • Tool calls — input, output, duration
  • Agent decisions — path taken and why
  • Errors, retries, fallbacks
  • Cost per run, per agent, per day

Cost management

  • Token budgeting — hard cap per run
  • Prompt caching — reuse repeated context
  • Model routing — use cheap model for simple tasks
  • Batching — group calls to reduce overhead
  • Right-sizing — don't use GPT-4o for every step

Tools

LangSmith (best DX), Langfuse (open source), OpenTelemetry for vendor-neutral tracing. Add from day one — you cannot debug multi-agent runs without traces.

Fine-tuning & Model Adaptation

When and how to adapt an LLM for your specific domain — a critical skill for AI engineers

  • Full fine-tuning — costs, compute, when it makes sense
  • Parameter-Efficient Fine-Tuning (PEFT) — adapting without full retraining

  • RLHF — Reinforcement Learning from Human Feedback

  • Datasets for fine-tuning — preparation, format, quality requirements
